## Directory Layout

```
.crew/
├── project.json                  # Project metadata
│
├── plan/                          # Hierarchical task structure
│   ├── README.md                 # Plan overview (generated by sync)
│   │
│   ├── 00-foundation/            # Epic directory: {number}-{slug}
│   │   ├── epic.json             # Epic metadata
│   │   ├── README.md             # Epic overview (generated)
│   │   └── tasks/                # Tasks within this epic
│   │       ├── 01-init-repo/     # Task directory: {number}-{slug}
│   │       │   ├── task.json     # Task metadata
│   │       │   └── README.md     # Task details (optional)
│   │       ├── 02-install-deps/
│   │       │   └── task.json
│   │       └── ...
│   │
│   ├── 01-features/              # Second epic
│   │   ├── epic.json
│   │   └── tasks/
│   │       └── ...
│   │
│   └── ...                        # Additional epics
│
├── setup/                         # Configuration and plan definition
│   ├── index.js                  # Main entry: hooks, checks, onInitCrew
│   ├── plan/                     # Plan definition (primary location)
│   │   └── index.js              # Exports createPlan(ctx) function
│   └── epics/                    # Epic-scoped resources
│       ├── prompts/              # Prompt templates (.md files)
│       └── executors/            # Executor scripts (.js files)
│
├── logs/                          # Task execution logs
│   └── {taskId}.log              # One log file per task execution
│
├── agents/                        # Claude agent definitions
│   └── {agent-name}.md           # Agent persona markdown
│
├── skills/                        # Claude skill definitions
│   └── {skill-name}/
│       └── SKILL.md              # Skill instructions
│
├── docs/                          # Project documentation
│   └── {doc-name}.md             # Markdown documents
│
├── goals/                         # Project goals
│   └── {goal-id}.json            # Goal definitions
│
└── progress.jsonl                 # Append-only progress log
```

## project.json

Root-level project metadata. Created by `crew init`.

```json
{
  "name": "My Project",
  "description": "A description of the project",
  "goal": "What the project aims to achieve",
  "epics": ["epic_abc123", "epic_def456"],
  "created": {
    "at": "2026-03-07T10:00:00.000Z",
    "by": "agent_system"
  },
  "updated": {
    "at": "2026-03-07T10:00:00.000Z",
    "by": "agent_system"
  }
}
```

| Field | Type | Description |
|-------|------|-------------|
| `name` | string | Project name |
| `description` | string | Project description |
| `goal` | string | High-level project goal |
| `epics` | string[] | Ordered list of epic IDs |
| `created.at` | ISO 8601 | Creation timestamp |
| `created.by` | string | Creator identifier |
| `updated.at` | ISO 8601 | Last update timestamp |
| `updated.by` | string | Last updater identifier |

## epic.json

Epic metadata. One per epic directory.

```json
{
  "id": "epic_abc123",
  "version": 1,
  "number": 0,
  "title": "Foundation",
  "status": "active",
  "task_ids": ["task_001", "task_002", "task_003"],
  "gates": [
    {
      "type": "plan",
      "required": true,
      "completed": true
    }
  ],
  "constraints": {
    "sequential": true
  },
  "created": {
    "at": "2026-03-07T10:00:00.000Z",
    "by": "agent_system"
  },
  "updated": {
    "at": "2026-03-07T10:00:00.000Z",
    "by": "agent_system"
  }
}
```

| Field | Type | Description |
|-------|------|-------------|
| `id` | string | Internal epic ID (`epic_xxx`) |
| `version` | number | Schema version |
| `number` | number | Epic number (0-indexed, used for display: M1, M2...) |
| `title` | string | Epic title |
| `status` | enum | `planned`, `active`, `done` |
| `task_ids` | string[] | Ordered list of task IDs in this epic |
| `gates` | Gate[] | Gates that must be passed |
| `constraints` | object | Execution constraints |
| `constraints.sequential` | boolean | Force sequential task execution |

### Epic Directory Naming

Epic directories use the pattern `{number}-{slug}`:
- `00-foundation`
- `01-features`
- `02-polish`

The number determines display order. The slug is derived from the title (lowercase, hyphens).

### Epic Status Transitions

```
planned → active → done
```

An epic becomes `active` when its first task starts, and `done` when all tasks complete.

## task.json

Task metadata. One per task directory.

```json
{
  "id": "task_001",
  "version": 1,
  "title": "Initialize Repository",
  "description": "Set up the project repository with standard tooling",
  "status": "done",
  "type": "coding",
  "assignee": "agent_builder",
  "epic_id": "epic_abc123",
  "dependencies": ["task_002"],
  "dependents": ["task_003"],
  "prompt": "Initialize a Next.js project with TypeScript...",
  "executorFile": "executor.js",
  "skills": ["repo/init"],
  "vars": {
    "framework": "nextjs"
  },
  "review": {
    "type": "human",
    "prompt": "Review the project structure"
  },
  "summaryPrompt": "Summarize what was created",
  "attempts": [
    {
      "number": 1,
      "started_at": "2026-03-07T10:05:00.000Z",
      "finished_at": "2026-03-07T10:06:30.000Z",
      "duration_ms": 90000,
      "success": true,
      "agent": "agent_builder"
    }
  ],
  "constraints": {
    "sequential": false,
    "blocks": ["task_005"]
  },
  "created": {
    "at": "2026-03-07T10:00:00.000Z",
    "by": "agent_system"
  },
  "updated": {
    "at": "2026-03-07T10:06:30.000Z",
    "by": "agent_builder"
  }
}
```

### Core Fields

| Field | Type | Description |
|-------|------|-------------|
| `id` | string | Internal task ID (`task_xxx`) |
| `version` | number | Schema version |
| `title` | string | Task title |
| `description` | string | Optional detailed description |
| `status` | enum | Current status (see below) |
| `type` | string | Task type: `coding`, `verify`, `planning`, `custom` |
| `assignee` | string | Agent assigned to this task |
| `epic_id` | string | Parent epic ID |

### Status Values

| Status | Meaning |
|--------|---------|
| `pending` | Not started, waiting for dependencies or scheduling |
| `active` | Currently being executed |
| `done` | Completed successfully |
| `failed` | Execution failed |
| `blocked` | Dependencies not met |
| `cancelled` | Removed from execution |
| `awaiting_review` | Completed, waiting for review gate approval |

### Status Transitions

```
pending → active → done
pending → active → failed
pending → blocked → pending → active → done
pending → active → awaiting_review → done
any → cancelled
```

### Dependency Fields

| Field | Type | Description |
|-------|------|-------------|
| `dependencies` | string[] | Task IDs this task waits for |
| `dependents` | string[] | Task IDs that wait for this task |

Dependencies are bidirectional — when task A depends on task B, task B lists A as a dependent.

### Instruction Fields

| Field | Type | Description |
|-------|------|-------------|
| `prompt` | string | Agent instructions for this task |
| `executorFile` | string | Path to executor script |
| `skills` | string[] | Skills to load for this task |
| `vars` | object | Task-scoped variables |

### Review Fields

| Field | Type | Description |
|-------|------|-------------|
| `review` | ReviewGate or ReviewGate[] | Review gate(s) |
| `review.type` | `human` or `agent` | Who reviews |
| `review.prompt` | string | Review instructions |
| `review.assignee` | string | Reviewer identity |
| `summaryPrompt` | string | Generate summary for reviewers |

### Attempt Tracking

Each execution attempt is recorded:

| Field | Type | Description |
|-------|------|-------------|
| `attempts[].number` | number | Attempt number (1-indexed) |
| `attempts[].started_at` | ISO 8601 | When execution began |
| `attempts[].finished_at` | ISO 8601 | When execution ended |
| `attempts[].duration_ms` | number | Execution duration |
| `attempts[].success` | boolean | Whether the attempt succeeded |
| `attempts[].agent` | string | Which agent executed |

### Task Directory Naming

Task directories use the pattern `{number}-{slug}`:
- `01-init-repo`
- `02-install-deps`
- `03-base-layout`

## progress.jsonl

Append-only log of every state transition. One JSON object per line. This is the crash-recovery mechanism — the system replays this log to reconstruct consistent state.

```jsonl
{"type":"task_status","taskId":"task_001","from":"pending","to":"active","at":"2026-03-07T10:05:00.000Z","by":"agent_system"}
{"type":"task_status","taskId":"task_001","from":"active","to":"done","at":"2026-03-07T10:06:30.000Z","by":"agent_builder"}
{"type":"epic_status","epicId":"epic_abc123","from":"planned","to":"active","at":"2026-03-07T10:05:00.000Z","by":"agent_system"}
```

### Entry Types

| Type | Fields | Description |
|------|--------|-------------|
| `task_status` | taskId, from, to | Task state transition |
| `epic_status` | epicId, from, to | Epic state transition |
| `plan_created` | planId | Plan was generated |
| `plan_reset` | planId | Plan was deleted |

### Recovery

On startup, the system:
1. Reads all entries from `progress.jsonl`
2. Replays state transitions in order
3. Compares with `task.json`/`epic.json` files
4. If inconsistencies found, the log is the source of truth

## Display ID Mapping

The `DisplayIdMapper` maps between internal IDs and user-facing display IDs:

| Internal Format | Display Format | Example |
|----------------|---------------|---------|
| `epic_xxx` | `M{number+1}` | `epic_abc123` → `M1` |
| `task_xxx` | `m{epicNum}.{taskNum}` | `task_001` → `m1.1` |

Display IDs are 1-indexed (M1, m1.1) while internal numbering is 0-indexed.

When communicating with users, always use display IDs. When calling store APIs or reading JSON files, use internal IDs.

## State Consistency

### Source of Truth Hierarchy

1. `progress.jsonl` — authoritative for state transitions
2. `task.json` / `epic.json` — authoritative for metadata
3. `README.md` files — derived views, regenerated by `crew sync`
4. `state.json` — aggregated view, regenerated by `crew sync`

### When State Gets Out of Sync

- Manual edits to JSON files without updating the progress log
- Crashes during write operations
- Concurrent access (not supported — single writer only)

**Fix**: Run `crew sync` to regenerate derived views. For deeper inconsistencies, the progress log takes precedence.

## Path Conventions

### Relative Paths

All paths in task definitions (inputs, outputs, prompt templates) are relative to the project root (`ctx.buildCtx.appDir`), not the `.crew/` directory.

### Epic Base Paths

When an epic has a `basePath`, template references in tasks are relative to that base:

```typescript
ctx.createEpic('setup', 'Setup')
  .basePath('./epics/0-setup')  // Templates resolve from here
```

### Executor Paths

Executor files (`executorFile` in task.json) are relative to the task directory:

```
.crew/epics/00-foundation/tasks/01-init/
├── task.json          # executorFile: "executor.js"
└── executor.js        # Resolved relative to this directory
```

### Log Paths

Log files are co-located with each task:

```
.crew/epics/<epic-slug>/tasks/<task-slug>/logs/<timestamp>-attempt-<N>.log
```
